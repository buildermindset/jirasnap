# I Needed This Tool, So I Built It: JiraSnap for VS Code

![JiraSnap — Capture Jira tickets in seconds, stay in your coding flow](../images/medium-header.png)

Most side projects start with an idea. This one started with a personal problem I was tired of repeating.

I would be deep in code, spot a bug or improvement, and think: "I should make a Jira ticket for this."

Then I would keep coding.

Then I would forget.

And once that happened enough times, I had to be honest with myself: the issue was not intention. The issue was friction.

## The Real Problem Was Context Switching

Yes, I could open Jira every time.

Yes, I could use existing Jira integrations.

But for the way I work, that was still too much context switching:

- Leave what I am typing
- Look to the left nav
- Find the Jira icon
- Click it
- Wait for it to load
- Start creating a ticket

That is enough delay for me to postpone it "for later."

And "later" often means "never."

## If It Is Not Built, Build It

I looked for something that matched exactly what I wanted:

- Stay in the editor
- Trigger with a command
- Capture the ticket in seconds
- Keep my flow intact

I could not find the right fit.

So instead of continuing the same bad habit and losing important work context, I did something about it: I built my own Visual Studio Code extension.

## What JiraSnap Does

JiraSnap is intentionally simple:

- Capture a Jira task directly from VS Code
- Add a quick note
- Include useful coding context (repo, branch, file, lines, timestamp)
- Return a clickable issue link

The goal is not to replace Jira. The goal is to remove enough friction that documentation actually happens.

## Why This Project Matters to Me

This solved a real day-to-day problem I have had for a long time.

It also gave me a chance to do something different and fun, and learn something new while building it.

That is one of my favorite parts of software engineering: when a practical need pushes you to create a tool that improves your own workflow.

And if it helps others too, even better.

## Giving Back

I have benefited for years from tools shared by other developers.

Open utilities, extensions, scripts, and small workflow hacks have saved me countless hours.

So this is my way of contributing back: I built the tool I needed, and I am sharing it in case it helps someone else stay in flow and still capture the work that should be tracked.
